# Architecture: ADEU Resident-Agent Connector Admission And External Assistant Bridge Family v0

Status: architecture / decomposition draft for `V62`.

Authority layer: architecture_decomposition.

This note does not authorize implementation by itself. It defines the bounded family
role that should admit one external connector path into the already shipped `V61`
governed communication membrane, without letting connector transport, connector
identity, or upstream assistant output become a hidden sovereign.

## 1. Family Role

`V62` is not another continuation family, not another resident communication family,
not the remote-operator family, and not yet the repo-widening family.

It is the family that owns:

- one bounded connector admission record over one selected connector path;
- one explicit principal-typing basis over admitted connector traffic;
- one bounded external assistant ingress bridge packet over admitted connector
  traffic in the starter slice;
- one bounded external assistant egress bridge packet back out through that same
  admitted connector path in a later same-family fill;
- one explicit connector provenance and exposure basis over that bridge;
- and later one advisory connector hardening / provenance-drift surface over that
  same selected bridge lineage.

The family therefore sits downstream of `V61` and upstream of any broader
connector-fleet, remote-operator, or execution-widening work.

- `V60` owns continuation / residual-intent identity.
- `V61` owns governed communication ingress / egress, interpretation, bridge-office
  binding, rewitness, and communication hardening over resident seams.
- `V62` owns connector admission and external assistant bridge posture over those
  already shipped `V61` packets.
- `V63` should later own remote operator UX and phone control-plane law.
- `V64` should later own repo-bound writable-surface authority.

## 2. Connector Boundary Thesis

The central law is:

- connector transport is transport only;
- connector identity is context, not permission;
- connector-carried principal identity is explicit and selected, not ambient;
- upstream external assistant output is bridge material, not native witness;
- upstream external assistant output is not charter compilation;
- upstream external assistant output is not act authority;
- and connector bridging must consume shipped `V61` communication law rather than
  inventing connector-native communication semantics.

So:

- raw connector payload is not native witness;
- raw connector payload is not `V60` task-law ingress;
- raw connector payload is not `V61` communication interpretation by itself;
- connector exposure is not admission by itself;
- human-via-connector semantics are not selected unless a later lock says so;
- and connector availability is not bridge office.

## 3. Starter Artifact Ladder

The family ladder should add these new shapes over the shipped `V61` surfaces:

1. `agentic_de_connector_admission_record@1`
   - typed admitted-connector artifact
   - explicit connector identity facts
   - explicit snapshot / exposure / route basis
   - explicit freshness basis
   - explicit selected principal class
   - explicit frozen-policy anchor
   - explicit admission verdict
2. `agentic_de_external_assistant_ingress_bridge_packet@1`
   - typed external-assistant ingress bridge artifact
   - explicit admitted connector ref
   - explicit selected connector principal posture
   - explicit consumed `V61` communication ingress or interpretation basis
3. later `agentic_de_external_assistant_egress_bridge_packet@1`
   - typed outbound bridge artifact back through the admitted connector path
   - explicit admitted connector ref
   - explicit selected connector principal posture
   - explicit consumed `V61` communication egress basis
4. later `agentic_de_connector_bridge_hardening_register@1`
   - advisory connector hardening / provenance-drift surface over the same admitted
     bridge lineage

Those shapes should consume, not reopen, the shipped:

- `agentic_de_communication_ingress_packet@1`
- `agentic_de_surface_authority_descriptor@1`
- `agentic_de_ingress_interpretation_record@1`
- `agentic_de_communication_egress_packet@1`
- `agentic_de_bridge_office_binding_record@1`
- `agentic_de_message_rewitness_gate_record@1`
- `agentic_de_governed_communication_hardening_register@1`

The starter slice is instantiated here only because `V63` is later.

That should be read as:

- instantiated_here:
  - one admitted external connector path
  - one external-assistant-only principal selection over that path
  - one bounded ingress bridge starter over shipped `V61` packets
- deferred_to_later_family:
  - remote operator UX and phone control plane
  - repo-bound writable authority
- deferred_to_later_lane:
  - explicit outbound bridge over the same admitted connector path
  - connector advisory hardening / provenance drift
  - broader connector-fleet admission
  - human-via-connector semantics

## 4. Starter Runtime Boundary Model

The starter connector path should be:

```text
already-real connector exposure / snapshot facts
  -> connector admission record
  -> selected principal over admitted connector traffic
  -> external assistant ingress bridge packet
  -> consume shipped V61 governed communication packet law
  -> later if bridge output is emitted:
       consume shipped V61 communication egress basis and shipped V61-B office /
       rewitness posture where relevant
       -> external assistant egress bridge packet
  -> later advisory connector hardening / provenance drift
```

What changes here is not continuation law, not resident communication law, not
bridge-office law, and not execution law.

What changes is that one external assistant path stops being ambient transport and
becomes one admitted, provenance-bearing bridge over shipped communication law.

## 5. Starter Freeze

`V62-A` should stay frozen to one exact starter path:

- one already-real connector path only
- one selected connector principal only:
  - external-assistant principal selected
  - human-via-connector principal deferred
- one connector admission record only
- one external assistant ingress bridge packet only
- no external assistant egress bridge packet yet
- one admitted connector identity / snapshot / exposure basis only
- no remote-operator UX
- no broader connector fleet
- no repo-bound writable authority
- no execute widening
- no dispatch widening

The family should not expand to phone control, remote operator command surfaces,
repo-bound write authority, execution widening, or delegated worker law in the first
slice.

## 6. Admission Law

Connector admission should remain explicit and replayable.

At minimum the starter family should preserve these laws:

- admission derives from:
  - typed connector identity facts
  - typed snapshot / exposure / route facts
  - typed freshness basis
  - selected principal class
  - frozen policy basis
- admission is not connector availability alone;
- admission is not ambient transport enablement;
- admission is replayable:
  - same connector facts
  - same exposure facts
  - same frozen policy
  - same admission verdict

Only the connector admission record may say that a connector path is selected here.

Snapshot visibility alone may not self-promote into admission.
Missing or stale snapshot / exposure basis must fail closed.

## 7. Bridge Law

External assistant bridging should remain explicit and bounded.

At minimum the starter family should preserve these laws:

- ingress bridge derives from:
  - admitted connector record
  - typed external assistant payload facts
  - shipped `V61` communication law
  - frozen policy basis
- later egress bridge derives from:
  - admitted connector record
  - shipped `V61` communication egress basis
  - shipped `V61-B` office / rewitness posture where relevant
  - frozen policy basis
- bridge packets are typed transport artifacts only;
- bridge packets are not native witness;
- bridge packets are not charter mutation;
- bridge packets are not continuation mutation;
- bridge packets are not act authority;
- bridge packets are replayable:
  - same admitted connector basis
  - same consumed communication basis
  - same frozen policy
  - same bridge packet

Only the bridge packets may define bounded external assistant ingress / egress for
the slice.

Connector payload text alone may not self-promote into lawful bridge semantics.
`V62-A` does not yet solve human-to-Codex via connector unless a later lock selects
that principal explicitly.

## 8. Non-Generalization Law

Connector evidence should remain path-level and non-generalizing by default.

At minimum the starter family should preserve these laws:

- one admitted connector path does not generalize by default into:
  - human-via-connector law
  - remote-operator law
  - broader connector-fleet trust law
  - repo-authority law
  - execute-authority law
  - dispatch law
- one successful external assistant bridge does not prove product-surface law;
- one admitted connector path does not mint standing authority for later connectors.

## 9. Later Family Boundary

`V62` should stop before:

- phone or remote operator control semantics;
- broad connector-fleet migration;
- repo-bound writable authority;
- execute widening;
- dispatch widening;
- delegated worker export.

Those are later branch or later family concerns.
