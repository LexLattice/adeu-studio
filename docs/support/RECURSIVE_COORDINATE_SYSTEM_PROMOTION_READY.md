## CURRENT_RULE_VS_UNIVERSAL_CLAIM_DISTINCTION

This is the critical cleanup.

The doctrine should now explicitly use three different verbs:

* **is** for structural axioms
* **constitutes** for the retained coordinate law
* **is currently authorized** for repo-constitutional posture

That prevents future readers from confusing ontology with current policy.

The most important rewrite is the operative rule.

Old hardening phrasing:
`operative_nodes_must_be_instance_level`

Promotion-ready phrasing:

* `family × operative` and `bounded × operative` remain **invalid in the retained grid**
* `meta × operative` is **reserved**
* `released repo operative surfaces are currently instance-bound`

That is materially cleaner. It says:

* some exclusions are constitutive of the current grid
* one exclusion is only current repo posture
* one cell is representable but still reserved

The repo evidence supports exactly that split. The current runtime operative surfaces are concrete state artifacts such as `execution_topology_state@1` and `write_lease_state@1` in `packages/urm_runtime/src/urm_runtime/orchestration_state.py:358-395`. By contrast, visibility and topology views explicitly declare themselves non-authoritative or read-only derived surfaces in `packages/urm_runtime/src/urm_runtime/worker_visibility.py:31-32,143-160` and `packages/urm_runtime/src/urm_runtime/topology_duty_map.py:32-34,113-117`. At the same time, the repo already names `recursive_governance_consumer` as a consumer class in `packages/adeu_repo_description/schema/repo_descriptive_normative_binding_frame.v1.json:62-70`, so higher-order recursion is representable; but the current meta-core explicitly forbids widening into recursive amendment or mutation doctrine in `docs/DRAFT_SCHEMA_META_CORE_v0.md:211-217`. That is why `meta × operative` is **reserved**, not **invalid**.

A second cleanup concerns promotion law.

`projection_may_not_mint_authority` should remain a retained coordinate law, not a merely contingent repo habit. The architecture family states it explicitly (`docs/ARCHITECTURE_ADEU_ARCHITECTURE_IR_v0.md:163-171,692-752,948-959`), and the descriptive-to-normative binding frame formalizes the same separation between descriptive artifacts and normative use (`packages/adeu_repo_description/schema/repo_descriptive_normative_binding_frame.v1.json:42-125`). So this is not a temporary style preference. It is a constitutive law of the current coordinate doctrine.

A third cleanup concerns implementation-era hygiene.

Rules like `support_qualifier max = 2`, current carrier leaf normalization, and current extractor precedence are useful, but they are not timeless ADEU truths. They should be tagged as contingent repo implementation assumptions so later toolchain evolution does not look like doctrinal rebellion.
