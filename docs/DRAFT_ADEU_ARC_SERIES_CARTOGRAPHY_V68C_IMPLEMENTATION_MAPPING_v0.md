Status: support note for the planned `V68-C` implementation pass.

Authority layer: support only.

This note does not authorize implementation by itself. It records the expected
`V68-C` slice shape early so the full `V68-A` / `V68-B` / `V68-C` ladder can receive
joint external review before implementation begins.

## Slice Thesis

`V68-C` should complete the `V68` family by hardening tool-applicability mapping and
recursive-coordinate posture over the cartography generated by `V68-A` and `V68-B`.

It should answer:

- which repo tools were applicable to which cartography claims?
- which tool results were namespace-limited, inconclusive, or not applicable?
- should recursive-coordinate records be emitted now, or should coordinate absence
  be represented deliberately?
- what bounded coordinate payload can later `V69` consume without treating the map
  as candidate admission?

`V68-C` does not begin `V69`. It prepares the evidence and coordinate posture that
`V69` may later consume.

## Consumed Basis

`V68-C` should consume shipped `V68-A` and `V68-B` outputs:

- the validated cartography object family;
- the populated namespace, closure, branch, support-lineage, and evidence indexes;
- the derivation manifest and gap scan shipped by `V68-B`;
- the selected current gap rows that remain open after `V68-B`.

It should also consume observed tool results from existing repo tools, including only
tools that are actually run or explicitly marked not run:

- arc-bundle lint, for starter / closeout namespace checks;
- closeout consistency lint;
- semantic compiler closeout lint;
- instruction policy view check;
- recursive-coordinate lints where applicable;
- repo-description derivation or test-selection tools where applicable;
- targeted pytest runs selected by the repo-native selector, where applicable.

## Selected `V68-C` Surfaces

`V68-C` should refine and populate existing `V68-A` surfaces:

- `repo_arc_mapping_tool_applicability_report@1`
- `repo_recursive_coordinate_emission_plan@1`

It should add one helper surface so observed tool runs remain independently
testable:

- `repo_arc_cartography_tool_run_evidence@1`

The helper surface should record observed command/tool runs and applicability
judgments only. It must not replace stop-gate, test, closeout, or release evidence.

## Tool Applicability Model

Each tool applicability row should carry:

- `tool_id`
- `tool_invocation_ref`
- `target_claim_id`
- `target_namespace_kind`
- `applicability_posture`
- `claim_horizon`
- `observed_result_ref`
- `limitation_note`
- `recommended_followup`

Required applicability values remain:

- `applicable_and_supporting`
- `applicable_with_warnings`
- `namespace_limited`
- `not_applicable_to_claim`
- `failed_or_inconclusive`
- `not_run`

The `arc-closeout-check ARC=67` lesson should be a canonical negative example: an
`ARC=67` tool argument success is not proof about family `V67`.

## Coordinate Posture Model

`V68-C` must decide between:

- `coordinate_emitted`
- `coordinate_absent_deliberate`
- `coordinate_absent_tool_not_applicable`
- `coordinate_absent_missing_source`
- `coordinate_absent_review_required`

Default expected closeout posture is `coordinate_absent_deliberate` unless `V68-B`
has produced enough settled rows to make emitted coordinates useful. Emitting no
coordinates is acceptable; silently omitting coordinate posture is not.

If coordinates are emitted, they must be bounded map coordinates only. They may
identify:

- family / slice / arc references;
- source and support lineage references;
- evidence surface references;
- branch posture references;
- future-family hypothesis rows.

They must not:

- admit a recursive candidate;
- evaluate candidate quality;
- ratify a future change;
- authorize integration;
- authorize commit / merge / release;
- project an operator UI;
- dispatch work.

## Required Validators

`V68-C` validators should prove:

- tool-run evidence records command, cwd posture, target namespace, observed output
  artifact, exit status, limitation note, and applicability judgment;
- every tool result has an explicit applicability posture;
- tool success cannot be attached to a claim outside the target namespace;
- not-run and not-applicable tools stay explicit rather than disappearing;
- coordinate absence is represented deliberately when no coordinate records are
  emitted;
- emitted coordinate rows reference known cartography rows;
- emitted coordinate rows cannot carry candidate-intake, ratification, release, or
  dispatch authority;
- unresolved `V68-B` gap rows either stay open or are marked with a lawful
  resolution posture.

## Mandatory Reject Cases

Reject fixtures should cover:

- a passing tool result used as global proof without namespace binding;
- a failed tool run omitted from the applicability report;
- coordinate absence represented as silent success;
- a coordinate row that admits `odeu_conceptual_diff_report@1` into released schema
  work;
- a coordinate row that authorizes `V69` implementation;
- a tool result that treats support docs as closeout evidence;
- unresolved gap rows silently dropped before family closeout.

## `V68` Family Closeout Preparation

`V68-C` is expected to be the last implementation slice in this family. Its lean
slice closeout should therefore prepare, but not replace, the full family closeout.

After `V68-C` implementation merges:

- update the `V68-C` slice closeout evidence on `main`;
- run the full `V68` family closeout bundle;
- mark `V68` closed only if `A`, `B`, and `C` outputs align;
- carry unresolved future-family hypotheses into `V69` planning only as mapped
  hypotheses, not as locks.

## Non-Goals

`V68-C` does not own:

- `V69` recursive candidate intake;
- admission of internally generated conceptual schemas;
- adversarial review classification;
- human ratification;
- contained integration or rollback;
- commit, merge, or release authority;
- operator-facing product workbench;
- multi-worker or multi-model dispatch.
