## Gap

The repo no longer lacks:

- one released worker-boundary bridge for delegated execution over `V48`;
- one released bounded constitutional artifact-checking family over `V55`;
- one released ODEU simulation family over `V51`; or
- one released bounded UX family over the `ux_*` contracts in `adeu_core_ir`.

The repo still lacks:

- one repo-owned family that governs the resident agent's externally effective loop at
  runtime rather than only its surrounding artifacts;
- one typed **resident-agent domain packet** that binds task, scope, authority frame,
  risk posture, and capability posture before action;
- one typed **resident-agent interaction contract** that distinguishes advisory output
  from write / execute / dispatch entitlement;
- one typed **action-proposal and membrane-gate path** that requires structured
  proposal-before-action rather than free transition from reasoning to effect;
- one typed **conformance / harvest path** that closes the loop after action and feeds
  governance evidence forward; and
- one released law that states clearly that hidden cognition remains unconstrained while
  boundary crossing, action formation, and post-action traces are governed.

So the repo still lacks a closed-loop resident-agent governance family between:

- high-order schema and authority law;
- bounded constitutional artifact checking; and
- live resident-agent runtime action.

No existing released family already owns that layer.

## Relationship To `V48`, `V51`, `V55`, And The Support Stack

`V48` asks:

- how should already accepted policy and scope bind delegated worker execution?

`V51` asks:

- how should ADEU own one bounded ODEU simulation kernel / API / browser ladder?

`V55` asks:

- how should a bounded promoted stack be checked for constitutional coherence?

The support stack asks:

- what family shape, membrane doctrine, morph law, and interaction semantics should
  govern a resident agent in an ADEU-native environment?

This new family asks:

- how should the resident agent's effective runtime loop be governed without pretending
  to govern hidden cognition?
- how should typed packet, contract, proposal, checkpoint, and conformance artifacts
  become the live control surfaces around resident action?
- how should write / execute / dispatch transitions remain gated by explicit authority,
  evidence, and consequence law?
- how should unresolved, insufficient-evidence, authority-missing, and not-evaluable
  states remain first-class rather than collapse into action anyway?

So this family may constrain:

- resident-agent action formation and boundary crossing;
- contract loading and per-turn runtime posture;
- gated tool / write / execute / dispatch transitions;
- post-action diagnostics, conformance, and harvest;
- future agentic-DE runtime consumers that need lawful interaction governance.

But it may not mint:

- control over hidden latent inference or token-level cognition;
- automatic CI or branch gating in the first slice;
- repo-wide replacement of existing worker-harness authority;
- direct product-surface entitlement by planning doc alone; or
- automatic promotion of support doctrine into released runtime law.

Planning relationship:

- `V48` remains the released worker-boundary family from `v31`;
- `V51` remains the released ODEU simulation family from `v34`;
- `V55` remains the released constitutional coherence family from `v38`;
- this draft introduces a fourteenth connected candidate family rather than widening
  any of those released families by ambient continuation.

Ordering discipline for this branch:

- `V55` checks constitutional coherence of admitted family surfaces;
- `V56` governs the resident-agent runtime boundary crossing and post-action trace;
- `V48` governs delegated worker execution after any dispatch is lawfully issued by
  `V56`;
- `V51` may shape packet, morph, and checkpoint semantics, but does not own runtime
  authority in this family by itself.

## Why This Path

- It addresses the real missing loop: the resident agent is still governed mainly by
  prompt prose plus surrounding tooling rather than by typed native runtime contracts.
- It avoids the wrong path of trying to formalize hidden cognition rather than
  boundary crossing.
- It reuses the strongest existing repo-native ideas:
  - packet loading;
  - interaction contract;
  - authority-boundary policy;
  - membrane gating;
  - conformance after execution.
- It leaves broad product rollout and CI gating later.
- It treats runtime governance as:
  - typed packet;
  - typed contract;
  - typed proposal;
  - bounded gate;
  - explicit post-action trace.
