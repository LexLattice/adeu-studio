## Purpose

Freeze the bounded writable-surface hardening posture for `V64-C` so the repo can
assess the shipped exact `V64-A` / `V64-B` writable-surface lineage for advisory
writable-surface hardening / provenance drift without silently widening
writable-surface selection law, lease issuance law, target-admission law,
restoration law, reintegration law, communication law, continuation law,
connector law, remote-operator law, broad repo-admin law, all-repo authority,
shell authority, execute authority, dispatch authority, delegated worker
authority, product authority, or hidden-cognition governance.

`vNext+178` authorizes docs plus the first `V64-C` implementation path over
existing repo-owned packages and thin backend seams, but not fresh
writable-surface selection, fresh lease issuance, fresh target admission, live
restoration / reintegration mutation, broader mutation-semantics widening,
all-repo authority, shell or terminal control, execute widening, dispatch
widening, delegated worker export or reconciliation, product/UI rollout as
authority, advisory-to-live promotion, or hidden-cognition governance.

In `vNext+178`, the new `V64-C` output is one advisory writable-surface
hardening surface only. It does not change `V64-A` writable-surface selection,
`V64-A` lease semantics, `V64-A` target-admission semantics, `V64-B`
restoration semantics, `V64-B` reintegration semantics, `V59` continuity
semantics, `V60` continuation semantics, `V61` governed communication semantics,
connector semantics, remote-operator semantics, or exit behavior by default.
