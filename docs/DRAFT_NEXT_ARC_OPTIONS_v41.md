## Gap

The repo no longer lacks:

- one released packet / checkpoint / ticket family for resident-agent action law;
- one released bounded observed local-effect family over a selected live path; or
- one released advisory family over post-effect hardening discussion.

The repo still lacks:

- one admitted live-turn history surface over the existing URM copilot path;
- one artifact-authoritative ticket-to-effect handoff inside that live path;
- one lawful reintegration report that closes a real resident turn over current-turn
  packet / proposal / checkpoint / ticket / observation evidence;
- and one way to keep outer harness capability surfaces necessary at most, never
  sufficient.

So the repo still lacks the slice that would make one real resident turn say:

- this live session admitted these exact turn facts
- this exact `V56` proposal / checkpoint / ticket lineage governed the act
- this exact `V57-A` observed effect happened under that lineage
- and the turn was closed on that basis rather than on ambient harness state

## Relationship To `V48`, `V55`, `V56`, And `V57`

`V55` asks:

- how should bounded promoted families be checked for constitutional coherence?

`V56` asks:

- how should the resident agent's externally effective loop be governed through packet,
  contract, proposal, checkpoint, ticket, conformance, and harvest?

`V57` asks:

- how should one already selected `V56` live path produce bounded observed effect,
  bounded restoration, and later bounded hardening evidence?

This new family asks:

- how should the existing live URM copilot session path be composed with the shipped
  `V56` / `V57` stack so one real resident turn can be governed by those artifacts
  rather than by ambient session capability alone?
- how should one current live turn externalize its admission, ticket-to-effect handoff,
  and reintegration without minting new action authority?

So this family may constrain:

- live-turn admission into explicit current-turn artifacts;
- harness transition from ticket lineage to observed effect;
- reintegration of observed effect back into the six-lane turn report;
- and artifact precedence for what counts as current-turn witness.

But it may not mint:

- new checkpoint semantics;
- new ticket semantics;
- new `local_write` semantics;
- new `local_reversible_execute` semantics;
- broader repo-write authority;
- delegated or external dispatch authority;
- worker execution already owned by `V48`;
- hidden-cognition governance;
- product/UI rollout as authority;
- or any advisory-to-live promotion from `V56-C` or `V57-C`.

Ordering discipline for this branch:

- `V55` checks constitutional coherence of the family surfaces;

- `V57` remains the sole owner of observed / restored / hardening evidence over the

- `V51` may shape membrane and reintegration reading, but does not own runtime
  authority here.

## Why This Path

- It addresses the actual missing capability:
  lawful composition of the shipped inner `V56` / `V57` action law with the already
  real outer URM live harness.
- It keeps the next family narrow:
  one current-turn path, one exact class, one exact write kind, one exact target root.
- It prevents the main sovereignty failure:
  `writes_allowed` or approval posture being misread as equivalent to `V56` ticket
  entitlement.
- It keeps current-turn witness explicit:
  transcript and event stream remain observability surfaces, not native witness.
- It defers breadth widening until the first live path is operationally real:
  no second `local_write` exemplar yet, no `local_reversible_execute`, no dispatch.
- It follows the practical harness doctrine:
  institutionalization should mean an artifact-authoritative state machine, not more
  prose.
