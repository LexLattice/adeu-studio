# Draft Next Arc Options v47

Status: planning draft after `docs/DRAFT_NEXT_ARC_OPTIONS_v46.md`, assuming
`V63-A` has landed on `main` through `v173`.

Authority layer: planning.

This draft does not automatically supersede `docs/DRAFT_NEXT_ARC_OPTIONS_v46.md`.
Instead, it records the next connected candidate after the bounded `V63-A`
starter:

how should ADEU add one richer typed remote command/control bridge over the
already admitted `V63-A` remote operator session and read-mostly view without
letting richer intervention become connector law, broad remote-admin law, repo
authority, execute authority, dispatch authority, or free-form shell control?

This is a planning document only. It is not a lock doc and does not authorize
runtime behavior, schema release, or implementation by itself.

## Baseline

- `V60-A` through `V60-C` remain the released continuation / residual-intent
  family on `main`.
- `V61-A` through `V61-C` remain the released governed communication family on
  `main`.
- `V62-A` through `V62-C` remain the released connector admission / external
  assistant bridge family on `main`.
- `V63-A` is treated here as the closed remote-operator starter:
  - one explicit `remote_operator` principal only
  - one admitted remote session / surface identity only
  - one read-mostly phone-safe view only
  - one tiny bounded response set only:
    - `acknowledge`
    - `approve`
    - `continue`
    - `pause`
    - `escalate`
- the family architecture and family mapping already place the next same-family
  widening at `V63-B`, not by reopening `V63-A` in place.

## Gap

The repo no longer lacks:

- one admitted remote operator session posture;
- one read-mostly phone-safe task / ask / evidence view over shipped `V60` /
  `V61` lineage; or
- one tiny bounded starter response set that explicitly consumes shipped law.

The repo still lacks:

- one typed richer remote command/control bridge over the admitted `V63-A`
  session;
- one explicit place for structured answers and clarifications to land without
  pretending they are just starter responses;
- one explicit place for inspect-rich intervention to land without becoming broad
  remote-admin law;
- one explicit bridge posture that remains downstream of shipped `V60`, `V61`,
  and closed `V63-A` session/view law rather than inventing a new sovereign
  command plane.

## Relationship To `V63-A`, `V64`, And `V65`

`V63-A` asks:

- how should one remote operator session be admitted, viewed, and given one tiny
  bounded response set over shipped `V60` / `V61` lineage?

This next slice would ask:

- how should one richer typed remote intervention bridge consume the already
  admitted `V63-A` session and shipped `V60` / `V61` basis?
- how should structured answers, clarifications, and inspect-rich interventions
  remain typed, replayable, and non-sovereign?
- how should richer intervention remain distinct from:
  - connector law still owned by `V62`
  - repo-bound writable authority reserved for `V64`
  - delegated execution / dispatch widening reserved for `V65`

So this next slice may constrain:

- one `V63-A` to `V63-B` lane handoff record;
- one typed governed remote control-bridge packet over the same admitted remote
  operator session;
- one bounded richer intervention vocabulary for:
  - structured answers
  - clarifications
  - inspect-rich controls

But it may not mint:

- new continuation law;
- new governed communication law;
- new connector law;
- broad remote-admin law;
- repo-bound writable authority;
- execute widening;
- dispatch widening;
- free-form terminal control;
- direct file editing authority.

## Recommended Next Path

- family remains: `V63`
- next concrete slice: `V63-B`
- recommended selector outcome:
  - keep `V63-A` closed as the admitted session/view starter
  - select `V63-B` as the next default candidate

Default next-slice posture:

- consume the shipped `V63-A` remote operator session record
- consume the shipped `V63-A` remote operator view packet
- consume shipped `V60` / `V61` lineage intact
- add one typed remote control-bridge packet only
- admit richer intervention classes only where explicitly selected:
  - `structured_answer`
  - `clarification`
  - `inspect_rich`
- keep richer intervention transport-bounded and non-mutating by itself:
  - not charter mutation
  - not residual mutation
  - not continuation mutation
  - not communication-law mutation
  - not bridge office
  - not act authority
- keep all richer intervention non-generalizing by default:
  - not connector law
  - not broad remote-admin law
  - not all-device or all-surface law
  - not repo authority
  - not execute authority
  - not dispatch authority
